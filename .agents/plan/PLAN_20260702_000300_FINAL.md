#### Known Mobile Issues

- **@dnd-kit touch sensor:** `TouchSensor` has `activationConstraint: { delay: 200, tolerance: 6 }` in Board.jsx (line 54). This prevents accidental drags while scrolling. Test that cards can still be dragged after a 200ms hold.
- **Gantt touch drag:** The `startDateDrag` and `startLink` functions use `mousedown`/`mouseup`. These won't fire on touch devices. Need to add `touchstart`/`touchend` equivalents.
  - **Fix if needed:** In `GanttView.jsx`, add touch event handlers:
    ```jsx
    const handleTouchStart = (e, task) => {
      const touch = e.touches[0]
      dateDragRef.current = { startX: touch.clientX, moved: false, task }
    }
    ```
- **DataSphere touch:** Three.js handles touch via the existing mouse-drag code? Check `DataSphere.jsx` line 128-156. Three.js `renderer.domElement` handles `touchstart`/`touchmove`/`touchend` by default when `controls` is used. If not, add touch event listeners.

### 1.3 Performance Testing

**Goal:** Ensure the app runs smoothly with >200 tasks.

**Method:** Create a seed SQL script or use the Supabase dashboard to bulk-insert 200+ tasks. Then measure:

#### Metrics

| Metric | Target | How to measure |
|---|---|---|
| Board initial render | < 2s from page load to task cards visible | Chrome DevTools Performance tab |
| Drag-drop responsiveness | < 100ms from drop to UI update | Console.time around updateTask |
| DataSphere FPS | > 20 fps with 1500 particles | `stats.js` or Chrome DevTools FPS meter |
| List view sort | < 50ms to re-render | React DevTools profiler |
| Gantt scroll | < 100ms between scroll and bar positions | Chrome DevTools Performance tab |
| Filter panel filter | < 50ms to filter 200 tasks | Console.time around useMemo |

#### Seed Data Script

Create `supabase/seed-bulk.sql`:

```sql
-- Generate 200 tasks spread across projects
do $$
declare
  v_project_id uuid;
  v_user_id uuid := (select id from auth.users limit 1);
begin
  for i in 1..200 loop
    -- Pick a random project (or null for shared board)
    v_project_id := (select id from public.projects order by random() limit 1);
    
    insert into public.tasks (
      title, description, status, priority, due_date, assignee, project_id, created_by, position
    ) values (
      'Performance test task #' || i,
      'This task was generated for performance testing. It contains some sample text for description rendering.',
      case when random() < 0.33 then 'todo'
           when random() < 0.66 then 'doing'
           else 'done' end,
      case floor(random() * 4)::int
           when 0 then 'P0' when 1 then 'P1'
           when 2 then 'P2' else 'P3' end,
      case when random() < 0.5 then now()::date + (random() * 30)::int
           else null end,
      case when random() < 0.7 then (select id from public.profiles order by random() limit 1)
           else null end,
      v_project_id,
      v_user_id,
      (random() * 100000)::int
    );
  end loop;
end;
$$;
```

#### Optimization Targets (if performance is poor)

| Issue | Fix |

| Gantt renders all tasks every frame | Add `useMemo` for the filtered/sorted ganttTasks array (line 156-163) |

2. Note the project ref (e.g., `abcdefghijklmnopqrst`)
3. Link the project:
   ```bash
   cd supabase
   supabase link --project-ref <PROJECT_REF>
   ```

## Viability Assessment: Final Polish Plan

### Strengths

1. **Incremental execution** — each phase is independent. If deployment is blocked (no Supabase account), testing can still proceed.
2. **Low implementation risk** — most tasks are verification, testing, and documentation. The only new code is touch event handlers for Gantt on mobile, which is a well-understood pattern.
3. **Quick wins available** — the console error sweep and task.md updates can be done in parallel with other work.

### Risks

1. **Cross-browser regressions are unpredictable** — Safari and Firefox may reveal CSS or JS issues not seen in Chrome. Budget 30 minutes per issue found.
2. **Performance testing is subjective** — ">200 tasks" is a vague target. If performance is poor, optimization time could double.
3. **Deployment complexity** — first-time Vercel/Netlify setup can have unexpected issues (build commands, env vars, redirect rules for SPA routing). Add `vercel.json` if needed:
   ```json
   {
     "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
   }
   ```
4. **Supabase production requirements** — email confirmation must be enabled, which changes the sign-up flow compared to local dev. Test this explicitly.

### Critical Path

The deployment phase has hard dependencies:
1. Supabase project must exist → apply migrations → get keys
2. Frontend build must succeed → deploy to Vercel
3. Env vars must be set → app boots correctly

These steps MUST be done in order and cannot be parallelized.

### Non-Critical Tasks (can be skipped if time is limited)

- [ ] Performance testing (>200 tasks) — nice-to-have if no observable lag
- [ ] Adding unit tests for GanttView, DataSphere — code is stable, risk is low
- [ ] Mobile Gantt touch drag — rarely used on phones
